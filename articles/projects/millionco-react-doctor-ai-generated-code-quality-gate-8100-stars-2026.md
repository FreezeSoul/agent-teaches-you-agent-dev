# React Doctor — AI 生成代码的质量门禁

> **TRIP 四要素**：
> - **T（目标用户）**：使用 AI 编程工具（Cursor/Claude Code/Codex等）的 React 开发者，尤其是企业团队
> - **R（核心成果）**：一键扫描代码库，输出 0-100 健康分 + 可操作诊断，将 AI 生成代码的问题率从「未知」变为「可控」
> - **I（技术洞察）**：AI 编程工具擅长实现但缺乏代码质量感知——React Doctor 通过规则化的质量门禁填补这个空白
> - **P（证据支撑）**：8,100 Stars，npm 下载量持续增长，被多个 enterprise 团队用于 CI/CD 流程

---

## 为什么关注这个项目

当企业大规模采用 AI 编程工具时，一个关键问题浮出水面：谁来为 AI 生成的代码质量负责？

PayPal 的案例揭示了一个重要洞察：

> "If you measure it, you impact it. If you tell a developer their success is based on what percentage of code was generated by AI, they'll just ask AI to write verbose functions."
> — Michelle Chance, Head of Developer Platforms, PayPal

这段话的隐含前提是：**AI 生成的代码需要被质量检查**。但问题是，传统的 lint 和 review 工具是为人类开发者设计的——它们假设代码是由理解上下文的工程师写的，而 AI 生成的代码往往是「语法正确但模式有问题」。

React Doctor 正是为这个场景设计的：它专门检测 AI 生成代码中的反模式，并且将检测结果变成一个可以量化的健康分。

## 核心功能

### 一键健康分扫描

```bash
npx -y react-doctor@latest .
```

输出示例：
```
Score: 75 (Great)
- State & Effects: 3 warnings
- Performance: 1 error
- Architecture: 2 warnings
- Security: 0 issues
- Accessibility: 1 warning
- Dead Code: 4 issues
```

评分标准：
- 75+：「Great」
- 50-74：「Needs work」
- <50：「Critical」

### AI Agent 规则教学

更重要的功能是 `install` 命令——它会教 AI 编程工具避免写坏代码：

```bash
npx -y react-doctor@latest install
# 会提示选择要安装规则的 Agent（Claude Code/Cursor/Codex等）
```

这相当于给 AI 编程工具装了一个「免疫系统」——让它从一开始就避免写反模式，而不是事后检测。

### GitHub Actions 集成

```yaml
name: React Doctor
on: [pull_request, push]

jobs:
  react-doctor:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - uses: millionco/react-doctor@main
        with:
          diff: main
          github-token: ${{ secrets.GITHUB_TOKEN }}
```

发现的问题会自动作为 PR 评论发布，并且暴露 `score` 输出用于后续步骤。

## 技术架构

### 规则体系

React Doctor 的规则覆盖六个维度：

| 维度 | 覆盖内容 | 示例规则 |
|------|---------|---------|
| State & Effects | useEffect 滥用、derived state | `no-cascading-set-state`, `no-render-in-render` |
| Performance | 不必要的重渲染、memo 缺失 | `rerender-state-only-in-handlers` |
| Architecture | 组件职责、代码组织 | `no-array-index-as-key` |
| Security | XSS、 dangerouslySetInnerHTML | `react/no-danger` |
| Accessibility | a11y 规范 | `jsx-a11y/no-autofocus` |
| Dead Code | 未使用代码、console.log | `no-console` |

### 框架自适应

规则会根据项目框架和 React 版本自动调整：

> "Rules toggle automatically based on your framework and React version."

这意味着 Next.js 项目和 React Native 项目的检测规则集不同，React 17 和 React 19 的规则也有差异。

### Linter 集成

如果项目已有 ESLint 或 oxlint 配置，React Doctor 会自动合并这些规则到扫描中：

> "If you have a JSON oxlint or eslint config (.oxlintrc.json or .eslintrc.json), its rules get merged into the scan automatically and count toward the score."

### suppression 支持

对于合理的例外情况，支持多种 suppression 方式：

```tsx
// 单行 suppression（推荐形式）
// react-doctor-disable-next-line react-doctor/no-cascading-set-state
const [localSearch, setLocalSearch] = useState(searchQuery);

// 多规则 suppression
// react-doctor-disable-next-line react-doctor/rerender-state-only-in-handlers, react-doctor/no-derived-useState

// Block suppression（适用于 JSX）
{/* react-doctor-disable-next-line react/no-danger */}
<div dangerouslySetInnerHTML={{ __html }} />
```

`--explain <file:line>` 命令可以诊断 suppression 为什么不生效（常见原因：相邻注释、链断裂）。

## 快速上手

### 基本扫描

```bash
npx -y react-doctor@latest /path/to/react-project
```

### 仅看分数

```bash
npx -y react-doctor@latest . --score
```

### 输出 JSON（用于程序处理）

```bash
npx -y react-doctor@latest . --json
```

### CI/CD 模式（失败于 warning）

```bash
npx -y react-doctor@latest . --fail-on warning
```

### 安装 Agent 规则

```bash
npx -y react-doctor@latest install --yes
```

## 局限性与适用边界

**适用场景**：
- React/Next.js/Vite/React Native 项目
- 使用 AI 编程工具（Cursor/Claude Code/Codex等）
- 需要在 CI/CD 中集成代码质量检查

**不适用场景**：
- 非 React 项目（需要不同的质量门禁）
- 纯后端 Node.js 项目

**需要注意的**：
- 8,100 Stars 是相对较新的项目，规则体系还在演进
- 作为一个专门面向 AI 生成代码的 linter，它的规则覆盖范围可能不如传统 linter 全面
