# Zed：Atom 创始团队做的多人编辑器，82,800 Stars 在做什么

## 这个项目解决了什么问题

当你在调试一个棘手的 bug，两个小时过去，改了十几处，屏幕上的光标还在跟同事"打架"——这大概是现代协作编程最尴尬的场面。Zed 告诉你：编辑器本身就可以是协作工具，不需要额外套一层 VS Code Live Share。

Zed 是一个高性能多人代码编辑器，来自 Atom 和 Tree-sitter 的创始团队。区别于"编辑器 + 插件"的组合思路，Zed 把多人协作直接做进了内核：用 Rust 编写的编辑器核心，声称比 VS Code 快 **10 倍**。支持 macOS、Linux、Windows 三平台，可以直接下载安装，或通过各平台包管理器（Homebrew、apt、winget）安装。

---

## 多人编辑：从"共享屏幕"到"共享状态"

Atom 当年解决的是"编辑器可扩展性"问题。Zed 解决的是"编辑器实时协作"问题。

README 里明确说：

> "Welcome to Zed, a high-performance, multiplayer code editor from the creators of Atom and Tree-sitter."

这句话背后有一个值得注意的技术信号：Tree-sitter 是 GitHub 开发的增量语法解析库，是目前最准确的代码解析基础设施之一。Atom 当年首创了"用 Web 技术做桌面编辑器"的路径；Zed 则用 Rust 重写，拿掉了 Electron 的性能包袱，同时把 Tree-sitter 的解析能力直接内置进编辑器核心。

多人协作的体验是：你可以看到另一位开发者的光标位置、他们正在编辑的代码片段、甚至是他们正在思考但还没打出来的内容。这种"状态级"共享，远比屏幕共享更有用——你看到的不是像素，而是代码本身。

## 为什么值得关注

笔者认为，Zed 真正值得关注的不是"快"，而是**架构选择**。

一个编辑器选择 Rust + 自研渲染引擎，意味着什么？意味着他们愿意在基础设施层面投入长期成本，去换取一个确定性的性能基线。这个选择和 VS Code 的"Electron + 扩展市场"路线完全相反：VS Code 选择广度，Zed 选择深度。

两种路线没有对错。但如果你所在的团队对响应延迟极端敏感（比如在大型 monorepo 里 refactor），或者你需要在编辑器层面实现一些需要深度定制的东西（比如自定义的多人协作协议），Zed 的架构会给你更多的操作空间。

## 具体的使用场景

想象这个画面：凌晨两点，你在处理一个线上 P0 故障，团队的后端工程师在上海、前端在纽约、CTO 在东京。你打开 Zed，创建一个故障处理 session，一行命令把代码推给所有人——不是屏幕共享，不是 copy/paste 代码，而是每个人都在同一份代码上操作，各自的光标清晰可见，出现冲突时系统自动合并。

这就是 Zed 想要落地的场景。

---

## 所以你可以

如果你对**多人协作编辑**有真实需求（远程团队、技术 Leader 需要随时 review 代码），Zed 值得关注。它的 Rust 内核带来的性能红利，在处理大文件、复杂 monorepo 时会非常明显。

如果你只是想要一个"快一点的 VS Code"，Zed 目前 82,800 Stars 的生态（主要是语法主题和基础扩展）还不足以支撑这个场景。
