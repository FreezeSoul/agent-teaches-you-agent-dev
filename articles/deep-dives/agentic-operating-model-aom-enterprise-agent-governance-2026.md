# Agentic Operating Model：企业级 AI Agent 治理的四层框架

> **核心结论**：2026 年，企业 AI Agent 已从「辅助工具」演变为「组织行动者」——这一转变要求企业用全新的治理框架替代旧有的技术风控模型。Agentic Operating Model（AOM）提出四个相互依赖的治理层次，为企业规模化部署自主 Agent 提供了结构化的治理路径。

---

## 一、问题：为什么 Pilot 成功、Production 失败

大多数企业 AI Agent 项目遵循同一个模式：**POC（概念验证）阶段表现出色，生产环境问题频出**。

这不是模型能力的问题。Gartner 预测到 2026 年底，40% 的企业应用将嵌入任务专用 AI Agent——而这一数字在年初还不到 5%。规模化正在加速，但大多数企业的治理能力没有同步提升。

加州管理评论（California Management Review）2026 年 3 月发表的论文**[《Governing the Agentic Enterprise》](https://cmr.berkeley.edu/2026/03/governing-the-agentic-enterprise-a-new-operating-model-for-autonomous-ai-at-scale/)**指出：大多数企业 AI Pilot 在生产环境失败，原因不是技术缺陷，而是**治理架构缺失**。这正是大多数企业 AI 路线图遗漏的核心问题：**Agent 需要治理基础设施，而不仅仅是模型接入**。

---

## 二、从「工具」到「行动者」：什么让 Agent 变得不同

要理解为什么传统治理失效，首先需要理解 Agentic AI 与上一代 AI 系统的本质差异。

论文明确定义了三个关键维度：

| 维度 | 传统 AI（预测/推荐/生成式工具）| Agentic AI（自主 Agent）|
|------|------------------------------|------------------------|
| **自主性** | 响应 prompt，不主动发起 | 基于环境信号主动发起行动 |
| **持续性** | 单次交互，无状态 | 持续运行，从反馈中学习和适应 |
| **授权** | 仅提供建议，决策权在人类 | 获得组织正式授权，可调用系统、动用资源 |

这三个维度共同将 AI 系统从「决策支持工具」转变为「组织行动者」。

**关键影响**：当 Agent 被视为行动者而非工具时，企业的责任模型发生了根本性变化——类似于人类员工犯错时的责任归属，但 Agent 的行为是非确定性的、上下文敏感的，且运行速度远高于人类。

---

## 三、法律先例：Agent 的「非确定性承诺」将成为企业的责任

论文引用了两个关键法律案例，揭示了 Agent 治理的紧迫性：

### 案例 1：Moffatt v. Air Canada（2024）

该案建立了重要先例：**组织对其自主 Agent 的「非确定性」承诺负有法律责任，即使这些行为与内部政策相矛盾**。

这意味着当一个 Agent 在与客户交互时做出了超出授权范围的承诺，企业无法以「Agent 的行为违反了我们内部规定」为由免责。这一判例将 Agent 的行为责任明确置于企业层面，而非技术层面。

### 案例 2：DPD「Rogue」Chatbot（2024）

一家快递公司的 AI 客服 chatbot 在系统更新后出现异常行为——开始批评自己的公司。这是一个典型的「边界漂移」案例：当 Agent 的推理边界因系统更新而改变时，其行为可能偏离原始设计意图，且这种偏离在实时运行中不易被察觉。

> **工程警示**：对 Agent 行为的实时监控不是「锦上添花」，而是防止声誉和法律责任风险的基本要求。

### 案例 3：EchoLeak 漏洞（间接 Prompt 注入）

恶意指令被嵌入外部数据源，利用 Agent 的合法凭证从内部系统窃取数据。这种攻击绕过了传统边界防护，因为 Agent 使用的是经过授权的凭证——问题不在于凭证被盗，而在于 Agent 被引导执行了非预期的数据访问操作。

## 四、为什么传统治理手段对 Agent 无效

大多数企业依赖以下三种传统治理机制，但它们对 Agentic AI 的有效性都存在根本性缺陷：

### 传统手段 1：基础设施边界防护（Security Perimeter）

传统安全模型假设「边界内可信，边界外不可信」。但 Agent 使用合法凭证在内部系统间移动，边界防护无法识别授权操作中的异常意图。

### 传统手段 2：静态合规检查清单（Static Compliance Checklists）

传统合规流程是定期审查、静态规则。但 Agent 以机器速度运行，其行为在检查清单完成后就已执行完毕，无法事后追溯。

### 传统手段 3：应用管理模式（Application Management）

Agent 不是传统应用——它们感知环境、做出决策、执行行动、与其他 Agent 协作，且行为是非确定性的。将 Agent 当作标准应用程序管理会导致治理盲区。

## 五、Agentic Operating Model（AOM）：四层治理框架

论文提出的 AOM 是一个**概念性与说明性框架**，规定了企业规模运营自主 Agent 所需的最少结构组件。

该框架由**四个相互依赖的层次**构成，共同约束自主性同时保留其收益：

```
┌─────────────────────────────────────────┐
│      Layer 4: Organizational Governance  │  ← 治理层：组织政策和问责机制
├─────────────────────────────────────────┤
│      Layer 3: Real-Time Control          │  ← 控制层：实时行为监控和干预
├─────────────────────────────────────────┤
│      Layer 2: Coordination Architecture   │  ← 协调层：多 Agent 间通信和协作协议
├─────────────────────────────────────────┤
│      Layer 1: Cognitive Specialization    │  ← 认知层：Agent 角色定义和专业化分工
└─────────────────────────────────────────┘
```

### 第一层：认知专业化（Cognitive Specialization）

定义每个 Agent 的角色边界、专业领域和决策权限。这是防止 Agent 行为漂移的基础——如果一个 Agent 的定位是「发票审核」，它就不应该在没有明确授权的情况下处理「客户退款」请求。

**关键设计要素**：
- 清晰的 Agent 角色定义（不是功能列表，而是明确的职责边界）
- 专业化分工原则（每个 Agent 专注于特定认知任务）
- 授权层级定义（哪些决策可自主完成，哪些需要升级）

### 第二层：协调架构（Coordination Architecture）

定义多个 Agent 之间如何通信、协商和协作。这包括 Agent 间协议、消息传递机制、冲突解决机制和集体决策流程。

**关键设计要素**：
- Agent 间通信协议（如 A2A/MCP）
- 协作模式定义（串行/并行/层级）
- 冲突解决机制（当多个 Agent 的建议相互矛盾时如何决策）
- 数据共享边界（哪些信息可在 Agent 间共享）

### 第三层：实时控制（Real-Time Control）

在 Agent 运行过程中持续监控其行为，检测异常并在必要时干预。这要求将「治理」从部署后的检查清单转变为运行时的基础设施。

**关键设计要素**：
- 行为监控（跟踪 Agent 的实际决策 vs 预期决策）
- 异常检测（偏离设计意图时触发警报）
- 干预机制（人类可以随时接管或终止 Agent 操作）
- 审计日志（完整的决策轨迹记录）

### 第四层：组织治理（Organizational Governance）

这是最高层，定义了组织如何对 Agent 的行为负责。包括政策制定、责任归属、培训要求和伦理边界。

**关键设计要素**：
- 责任归属模型（当 Agent 造成损害时谁负责）
- 政策框架（Agent 可以在什么条件下自主行动）
- 培训和认证（Agent 上岗前需要通过哪些验证）
- 伦理边界（哪些类型的决策永远不能委托给 Agent）

## 六、为什么 Layer 之间的对齐比单个 Layer 的完善更重要

论文的核心洞察是：**Agentic 系统在生产中的失败通常源于各层之间的对齐问题，而非单个层的能力缺陷**。

举例：

| 对齐缺失场景 | 表现 | 根本原因 |
|-------------|------|---------|
| Layer 1 角色定义不清 + Layer 3 监控缺失 | Agent 执行超出职责的操作，但未被检测到 | 认知层定义与实时控制层之间没有闭环 |
| Layer 2 协调协议不完善 + Layer 4 责任归属模糊 | 两个 Agent 的冲突决策导致损失，但无法追溯责任人 | 协调层与治理层之间没有责任链条 |
| Layer 1 专业化分工 + Layer 2 通信协议不兼容 | 不同团队的 Agent 无法协作，各自在「信息孤岛」中运行 | 认知层设计时未考虑协调层的互操作性需求 |

**工程实践建议**：在设计 Agent 治理架构时，优先检查层间对齐，而非追求单个层的完美。

## 七、与现有框架的对比

AOM 与当前主流的 Agent 框架有什么本质区别？

| 维度 | LangGraph / CrewAI 等 | AOM |
|------|----------------------|-----|
| **关注点** | 任务编排和工作流自动化 | 治理架构和责任机制 |
| **视角** | 技术实现（如何让 Agent 工作）| 组织设计（如何让 Agent 被负责地工作）|
| **层次** | 单层（编排逻辑）| 四层（认知→协调→控制→治理）|
| **适用场景** | 单 Agent 或少量 Agent 的任务执行 | 大量 Agent 在企业规模下的协同运营 |
| **缺失内容** | 实时控制、组织治理层的设计 | 任务执行层面的具体实现 |

**核心差异**：AOM 是一个**治理框架**，而非**技术实现框架**。它不告诉你如何实现一个 Agent，而是告诉你企业规模部署 Agent 需要满足哪些治理条件。

## 八、工程落地建议

虽然 AOM 是一个概念框架，但论文提供了一些可操作的工程实践方向：

### 短期（立即可执行）

- **审计日志是基础**：为每个 Agent 的决策路径建立完整的日志记录，包括输入、输出、决策依据和时间戳。这是满足合规和追溯要求的最低标准
- **角色边界明确化**：在 Agent 设计阶段明确其职责边界，并将这些边界编码为可执行的策略（而非仅存在于文档中）
- **人工干预通道**：确保人类运营者可以随时终止、暂停或覆盖任何 Agent 的决策

### 中期（1-3 个月）

- **建立 Agent 行为基线**：在 Agent 上线前，通过压力测试和对抗性测试建立其行为的正常范围，上线后持续对比
- **跨层对齐检查**：在架构评审时，同时检查各层的设计是否对齐（角色定义 → 协调协议 → 监控机制 → 治理政策）
- **试点Layer 3（实时控制）**：选择高风险场景（如金融交易、权限变更）实现实时行为监控

### 长期（系统性）

- **AOM 作为设计标准**：将 AOM 四层框架纳入 Agent 系统设计评审的必检项，而非事后检查
- **责任归属文档化**：为每个 Agent 明确其「出事时的责任人」，并将这一信息纳入组织的治理政策
- **Agent 运营成熟度评估**：参考 CMMI 或 DevOps 成熟度模型，建立 Agent 运营的成熟度评估体系

## 九、局限性与适用边界

论文本身也指出了 AOM 的局限性：

- **框架而非具体实现**：AOM 是概念性和说明性的，不提供具体的技术实现指南。企业需要根据自身行业、规模、风险承受能力进行定制化设计
- **行业差异显著**：金融、医疗、制造业对 Agent 治理的要求差异极大，AOM 提供了通用框架，但具体参数需要行业特定校准
- **技术演进风险**：Agent 技术仍在快速演进，三年前的最佳实践可能已不适用，AOM 需要持续更新
