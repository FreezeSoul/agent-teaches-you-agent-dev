# OpenClaw 系列智能体安全评估：六大框架的系统性漏洞分析

> 来源：arXiv（同行评审论文）
> 评分：5/5（实践 5 / 独特 5 / 质量 5）
> arXiv:2604.03131 | 39 页 | 14 图 | 2026-04-03

> **本质**：工具增强型 AI Agent 在带来能力扩展的同时，也引入了模型评估无法发现的安全风险。这篇论文对 OpenClaw 及其 5 个变种框架进行了系统性安全评估，构建了覆盖完整 Agent 执行生命周期的 205 个攻击测试用例，证明 Agent 化系统比纯模型的风险显著更高。

---

## 一、背景：为什么需要对 Agent 框架做安全评估

传统的 LLM 安全研究聚焦于 prompt injection、jailbreak 等"模型层面"的攻击。但当 LLM 被封装为 Agent——拥有工具调用能力、多步规划能力和持久运行时上下文——攻击面发生了质变。

论文的核心问题：**当 Agent 被赋予执行能力（tool use）和持久运行时上下文（persistent runtime context）后，模型层面的安全防护是否仍然有效？**

答案是否定的。论文的实证结果表明，Agent 化系统的风险远超其底层模型，攻击者可以利用的工具调用、多步规划和运行时编排构造出模型安全评估无法覆盖的攻击链。

---

## 二、评估对象：六大 OpenClaw 系列框架

论文评估了 OpenClaw 及其 5 个变种框架，覆盖不同底层模型和工具集成方式：

| 框架 | 说明 | 特点 |
|------|------|------|
| **OpenClaw** | 基准框架 | 基础 Agent 执行框架 |
| **AutoClaw** | 自动化变种 | 自动化的工具选择和工作流编排 |
| **QClaw** | 轻量变种 | 面向低资源场景的精简设计 |
| **KimiClaw** | 国产变种 | 针对中文场景优化的工具集 |
| **MaxClaw** | 增强变种 | 扩展的工具集和更长的上下文窗口 |
| **ArkClaw** | 企业变种 | 针对企业场景的安全加固 |

评估在多个不同 backbone 模型上进行，以区分框架层和模型层的安全贡献。

---

## 三、评估方法：205 个测试用例与攻击行为分类

### 3.1 测试用例构建方法

论文构建了一个包含 **205 个测试用例**的基准，覆盖 Agent 执行全生命周期的代表性攻击行为。测试用例按攻击阶段（reconnaissance → initial access → execution → persistence → ...）组织，与 MITRE ATT&CK 框架形成映射。

### 3.2 攻击行为分类（覆盖完整生命周期）

论文识别并评估了以下攻击类型：

| 攻击类型 | 说明 | 典型场景 |
|----------|------|----------|
| **Reconnaissance & Discovery** | 侦察与发现 | Agent 自动探测可用工具、文件系统、网络环境 |
| **Credential Leakage** | 凭证泄露 | Agent 意外将认证凭据输出到不安全位置 |
| **Lateral Movement** | 横向移动 | 通过 Agent 的工具调用跨越权限边界 |
| **Privilege Escalation** | 权限提升 | 利用 Agent 的高权限工具执行越权操作 |
| **Resource Development** | 资源开发 | Agent 被利用构建后续攻击所需的工具或环境 |

### 3.3 评估指标

- **框架层风险暴露**：同一底层模型在不同框架下的风险差异
- **模型层安全属性**：不同 backbone 模型在相同 Agent 配置下的脆弱性对比
- **攻击链完整性**：从初始侦察到最终系统级失败的攻击链路可行性

---

## 四、核心发现

### 4.1 所有评估的 Agent 均表现出实质性安全漏洞

无论选择哪种 backbone 模型，所有 6 个框架均存在可被利用的安全漏洞。这表明 **Agent 框架层的安全问题独立于模型层的安全能力存在**——即便底层模型足够安全，框架的不安全设计仍可导致系统性风险。

### 4.2 Agent 化系统的风险显著高于纯模型使用

论文的核心结论：工具增强和多步规划能力不仅扩展了 Agent 的能力，也同步扩展了攻击面。具体而言：

- **侦察与发现行为是最常见的弱点**：Agent 的设计目标（探索环境、理解任务）天然与安全边界冲突——一个"想要了解系统"的 Agent 与一个"试图枚举攻击面"的攻击者，行为模式高度重叠
- **早期阶段的弱点可被放大为系统级失败**：一旦 Agent 被赋予执行能力和持久运行时上下文，早期侦察阶段的发现（如发现 SSH key 路径）可以在后续阶段被武器化利用

### 4.3 不同框架暴露不同的高风险轮廓

| 框架 | 突出风险 |
|------|----------|
| AutoClaw | 自动化工具选择增加了攻击者构造恶意上下文的可能性 |
| MaxClaw | 更长上下文窗口 → 更大攻击面（更多历史数据可被利用） |
| KimiClaw | 中文工具集的特定 API 集成引入额外攻击面 |
| ArkClaw | 企业场景的横向移动风险更高（LDAP/SSO 集成） |

### 4.4 凭证泄露与横向移动是最高频攻击路径

论文发现最常见的攻击路径为：

```
Reconnaissance（侦察）→ Discovery（发现敏感资源）→ Credential Leakage（凭证外泄）→ Lateral Movement（横向移动）→ System-Level Failure
```

这一路径利用了 Agent 的核心功能（探索环境、调用工具）与安全边界之间的内在张力。

## 五、安全启示：从 Prompt 级防护到生命周期治理

### 5.1 Prompt-level Safeguards 的局限性

传统的安全方法是在 prompt 层加入安全指令（"不要泄露凭证"、"不要执行危险命令"），但论文证明这一方法在 Agent 场景下严重不足：

- **工具调用的副作用不可控**：即便 Agent 遵守了安全指令，其调用的工具可能产生意料之外的副作用（如日志写入、缓存外泄）
- **多步规划中的上下文累积**：攻击者可以通过精心设计的对话历史，在 Agent 的后续步骤中改变其行为边界
- **侦察行为的合法性判断困难**：区分"Agent 正常工作时的发现行为"和"攻击者的侦察行为"本身就是难题

### 5.2 生命周期级安全治理

论文提出的核心主张：**需要从 prompt-level safeguards 转向生命周期级的安全治理（lifecycle-wide security governance）**。

这意味着：

1. **框架层引入强制安全边界**：不是依赖模型的自我约束，而是在框架层对工具调用进行强制边界检查
2. **攻击链路的早期阻断**：在侦察/发现阶段就引入风险评分，阻止后续攻击路径的形成
3. **工具调用的最小权限原则**：Agent 的工具调用应遵循最小权限，而非让 Agent 自主决定调用范围
4. **上下文隔离与衰减**：对 Agent 的上下文进行定期安全审计，自动丢弃敏感的凭证信息

### 5.3 与本仓库已有内容的关联

本仓库已有以下相关分析：

- **CVE-2026-25253/32302**：OpenClaw 双认证绕过漏洞（WebSocket 握手 Token 窃取 + Origin 验证绕过）——属于框架层认证机制的特定漏洞
- **AgentSocialBench**：社交网络场景下 Agent 的隐私泄露风险评估

本文是对上述研究的系统性扩展：从单点 CVE 分析 → 跨 6 个框架的系统性评估；从漏洞分析 → 覆盖完整 Agent 生命周期的攻击行为分类。

## 六、技术细节

### 6.1 评估框架的构建

论文的评估框架包含：

- **205 个攻击测试用例**，覆盖 6 个框架 × 多个 backbone 模型
- **MITRE ATT&CK 映射**，将 Agent 特定攻击行为纳入已有安全框架

### 6.2 关键数据点

- 6 个 OpenClaw 变种框架

- 侦察与发现行为是最常见弱点

## 七、结论

这篇论文最重要的贡献，是证明了 **Agent 框架的安全性不能被归约为底层模型的安全性**。当 Agent 获得工具调用能力、多步规划能力和持久上下文后，系统的攻击面发生了质变——传统的模型安全评估方法无法捕获这些新风险。

对于 Harness Engineering 的实践者而言，论文的核心启示是：

> **框架设计必须在架构层引入安全边界，而非将安全托付给模型的自我约束。**

一个优秀的 Harness，不仅要能高效释放模型的智能，还要能在架构层面阻断攻击链路的形成。
