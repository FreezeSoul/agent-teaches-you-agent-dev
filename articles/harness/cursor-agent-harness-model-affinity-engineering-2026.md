# Cursor Agent Harness 的模型亲和性工程：为什么同一个 Harness 无法适配所有模型

**本文核心论点**：Cursor 的工程实践揭示了一个关键认知——**Harness 不是模型无关的抽象层，而是需要针对每个模型进行深度定制的产品**。从工具格式（patch vs string replacement）到提示词风格（literal vs intuitive），从错误分类阈值到 mid-chat 切换处理，每一个模型都有独特的「性格」，而 Harness 的质量取决于对这些性格特征的理解和适配程度。

**一手来源**：[Cursor Blog: Continually improving our agent harness](https://cursor.com/blog/continually-improving-agent-harness)（2026-04-30）

**前置知识**：建议先阅读 [Anthropic「Building Effective Agents」](https://www.anthropic.com/engineering/building-effective-agents)，了解 Agent Harness 的基础概念

## 一、问题的起源：为什么模型需要不同的工具格式

当 Cursor 开始支持多个模型时，他们面临一个看似简单的问题：应该给每个模型配什么样的文件编辑工具？

直觉的答案是「统一」——所有模型使用相同的工具格式，通过 Prompt 指导模型如何调用。但这恰恰是错误的做法。Cursor 发现了关键事实：

> "OpenAI's models are trained to edit files using a patch-based format, while Anthropic's models are trained on string replacement. Either model could use either tool, but giving it the unfamiliar one costs extra reasoning tokens and produces more mistakes."
> — [Cursor Blog](https://cursor.com/blog/continually-improving-agent-harness)

这背后的机制是**模型的工具调用能力是在训练阶段形成的**。模型的工具调用本质上是「根据工具描述生成对应格式的调用参数」，但训练时的数据分布决定了它对不同格式的熟悉程度：

| 模型族 | 训练时的编辑格式 | 换用其他格式的成本 |
|--------|----------------|-----------------|
| OpenAI (GPT 系列) | Patch/diff 格式 | 额外的推理 token + 更高的错误率 |
| Anthropic (Claude 系列) | String replacement | 同样的代价 |

这不是模型「不会」，而是「不习惯」。就像一个习惯用右手写字的人被要求用左手——虽然能写，但速度慢且容易出错。

**工程启示**：在为 Agent 选择工具格式时，应该优先匹配模型训练时的格式偏好。如果必须使用非偏好格式，需要额外增加 Prompt 说明和示例，且仍会有推理效率损失。

## 二、更深的差异：提示词风格的适配

工具格式只是表面。Cursor 发现不同模型对提示词风格有截然不同的反应：

> "OpenAI's models tend to be more literal and precise in their instruction following, whereas Claude is a bit more intuitive and more tolerant to imprecise instructions."
> — [Cursor Blog](https://cursor.com/blog/continually-improving-agent-harness)

这个差异有深刻的训练背景：

**OpenAI 模型**：训练数据中高比例的代码和指令性文本，使其擅长精确执行明确指定的任务。当指令模糊时，它倾向于严格遵循字面意思，可能偏离预期结果。

**Anthropic 模型**：RLHF 训练中融入了更多的「理解意图」能力，使其在指令不精确时能推断合理方向，但同时也更容易产生「幻觉性遵从」——看似合理但实际错误的执行。

**工程实践中的影响**：

```
OpenAI 模型 Harness 设计：
- 指令需要精确、无歧义
- 工具描述要完整，参数边界要明确
- 避免长而复杂的复合指令

Anthropic 模型 Harness 设计：
- 允许一定的指令模糊度
- 可以用高层目标而非精确步骤描述任务
- 需要额外的边界检查机制
```

> 笔者认为：这个差异意味着**没有万能的 Agent 系统 Prompt**。为每个模型定制 Prompt 不是过度工程，而是必要的产品决策。

## 三、模型特有的「症状」：上下文焦虑（Context Anxiety）

最有趣的发现是当 Cursor 遇到一个模型的特殊行为——他们称之为「上下文焦虑」：

> "For example, we observed one model develop what we came to call context anxiety: As its context window filled up, it would start refusing work, hedging that the task seemed too big."
> — [Cursor Blog](https://cursor.com/blog/continually-improving-agent-harness)

这不是 Bug，而是模型在特定上下文状态下的行为模式。Cursor 通过**提示词调整**（prompt adjustment）减轻了这一行为。

**上下文焦虑的机制推测**：

1. 模型在预训练中可能遇到过「上下文耗尽导致任务失败」的样本
2. 当检测到上下文利用率高时，激活了「保守策略」模式
3. 表现为拒绝接受任务或要求用户精简输入

**应对策略的工程价值**：

- 不能依赖模型「自己调节」——需要在 Harness 层感知上下文状态
- 提示词调整比模型重新训练更高效——这是 Harness 层的价值
- 需要在测试中专门覆盖「高上下文利用率」场景

> 笔者认为：「上下文焦虑」揭示了一个重要的设计原则——**Harness 需要对模型的非理性行为（从人类视角看）有足够的预判和补偿机制**。这不是让模型更聪明，而是让 Harness 更懂模型。

## 四、Mid-Chat 模型切换：Harness 最难的挑战之一

当用户在对话中途切换模型时，Harness 面临双重挑战：

### 挑战 1：历史上下文与模型训练分布不匹配

> "When a user switches models, Cursor automatically switches to the appropriate harness, with that model’s customized set of prompts and tools. However, the model still has to apply those tools to a conversation history that was produced by a different model and is out of distribution from what it was trained on."
> — [Cursor Blog](https://cursor.com/blog/continually-improving-agent-harness)

对话历史是「另一个模型生成的」，包含该模型的调用模式、输出风格、甚至潜在的错误模式。新的模型需要将这个历史作为输入理解，但这超出了它的训练分布。

### 挑战 2：缓存失效的性能惩罚

> "Caches are provider- and model-specific, so switching means a cache miss and a slower, more expensive first turn."
> — [Cursor Blog](https://cursor.com/blog/continually-improving-agent-harness)

缓存穿透不仅影响性能，还影响成本。Harness 需要在这种情况下做出权衡。

### Cursor 的缓解方案

**方案 1：切换提示词**

```python
# 伪代码示例
CUSTOM_INSTRUCTIONS = """
You are taking over mid-chat from a different model.
The conversation history below was generated by a different model.
Ignore any tool calls in the history that are not in your current tool set.
"""
```

**方案 2：对话摘要（实验性）**

在切换时对对话历史进行摘要，减少 token 消耗并提供一个「清洁的开始」。但风险是丢失关键细节。

**方案 3：Subagent（推荐方案）**

> "Another way to sidestep the challenges of mid-conversation model switching is to instead use a subagent, which starts from a fresh context window."
> — [Cursor Blog](https://cursor.com/blog/continually-improving-agent-harness)

Subagent 从全新的上下文开始，避免了历史分布问题，但也失去了对话连续性。这是个权衡，不是银弹。

**工程建议**：
- 除非有明确理由，不建议在复杂任务中途切换模型
- 如果必须切换，优先使用 Subagent 模式而非对话延续
- 切换后的第一个 Prompt 应该包含「接管说明」，帮助模型理解上下文

## 五、模型定制化的层次结构

Cursor 的实践经验揭示了一个**多层定制化**的体系：

| 层次 | 定制内容 | 变更频率 |
|------|---------|---------|
| **工具格式** | Patch vs String Replacement | 模型架构级，几乎不变 |
| **提示词风格** | Literal vs Intuitive | 微调级别，可能随版本变化 |
| **参数阈值** | 上下文焦虑触发条件 | 实验驱动，快速迭代 |
| **错误分类标准** | 哪些错误率触发报警 | 产品反馈驱动 |

每一层有不同的时间常量和决策流程。

**工具格式是架构决策**——一旦选定很难改变，需要在模型选型时确定。

**提示词风格是产品决策**——需要大量 A/B 测试和用户反馈。

**阈值和分类是运维决策**——需要监控和告警系统支撑。

> 笔者认为：这个层次结构解释了为什么「做一个通用的 Agent Harness」是个伪命题。不同层次的决策有不同的约束条件，用同一套流程处理所有层次必然导致低效或质量损失。

## 六、错误分类的模型亲和性

Cursor 将工具调用错误分为两类：

- **Unknown Error**：真正的 Bug，必须修复
- **Expected Error**：模型在合理范围内犯的错，如：
  - `InvalidArguments`——模型给出了格式正确但内容不合理的参数
  - `UnexpectedEnvironment`——上下文窗口中的矛盾（如文件说不存在但实际存在）
  - `ProviderError`——供应商服务中断

> "We compute baselines per-tool and per-model, because different models may mess up tool calls at different rates."
> — [Cursor Blog](https://cursor.com/blog/continually-improving-agent-harness)

这意味着**判断「正常错误率」的标准必须因模型而异**。同一个工具，Claude 的错误率可能低于 GPT，反之亦然。如果用统一阈值，必然导致大量误报或漏报。

**工程实践**：

```python
# 伪代码
ERROR_BASELINES = {
    "gpt-5": {"edit_file": 0.02, "grep": 0.05, "bash": 0.01},
    "claude-4": {"edit_file": 0.01, "grep": 0.03, "bash": 0.015},
    # 每个模型+工具组合都有独立的基准线
}
```

## 七、未来方向：Harness 的核心地位

Cursor 明确指出：

> "The ability to orchestrate that kind of coordination will live in the harness rather than any single agent. This means that, while harness engineering has always been important for agent success, it's only going to be more critical going forward."
> — [Cursor Blog](https://cursor.com/blog/continually-improving-agent-harness)

在多 Agent 系统中，Harness 的职责从「管理单个 Agent」扩展到「协调多个 Agent」。这意味着：

1. **调度决策**：哪个 Agent 处理哪个子任务
2. **格式适配**：不同 Agent 可能使用不同的工具格式
3. **结果整合**：将多个 Agent 的输出合并为一致的结果

> 笔者认为：随着 Agent 能力提升，Harness 的核心价值从「让单个 Agent 正确工作」转向「让多个 Agent 协作工作」。这是一个质变——从工具层到协调层的跨越。

## 八、核心工程结论

| 结论 | 含义 |
|------|------|
| 模型有「性格」 | 工具格式、提示词风格、错误模式都有模型特异性 |
| Harness 需要分层定制 | 架构决策 vs 产品决策 vs 运维决策，不同层次用不同流程 |
| 错误阈值必须因模型而异 | 通用阈值会导致大量误报/漏报 |
| Mid-chat 切换是高风险操作 | 优先使用 Subagent 模式而非对话延续 |
| 多 Agent 时代 Harness 是协调层 | 价值从「单个 Agent 管理」到「多 Agent 协调」 |

**实践检查清单**：

```
[ ] 是否为每个模型配置了训练偏好对应的工具格式？
[ ] 是否针对不同模型调整了提示词风格（literal vs intuitive）？
[ ] 是否为高上下文利用率场景设计了专门的处理机制？
[ ] 错误报警阈值是否按模型+工具组合分别设置？
[ ] Mid-chat 模型切换是否默认使用 Subagent 而非对话延续？
[ ] 是否为多 Agent 调度预留了 Harness 扩展接口？
```
