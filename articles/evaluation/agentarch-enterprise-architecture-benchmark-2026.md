## 一句话总结

AgentArch 首次系统评估了 18 种 Agent 架构配置在 6 个主流 LLM 上的表现，发现：**最优架构因模型而异，不存在万能设计**；在企业任务上，即使是表现最好的模型，复杂任务成功率也仅 35.3%。

## 背景与动机

### 现有评估的碎片化问题

之前的研究要么单独评估 Agent 的某个组件（工具调用、记忆、多 Agent 协作），要么在游戏/研究这类非企业场景下评估。这种碎片化导致：

- **缺乏组合效应的理解**：Orchestration × Memory × Prompting Style 三者如何交互？不知道。
- **缺乏企业场景聚焦**：企业工作流的特点是步骤必须按固定顺序执行、需要与现有系统集成、对可靠性要求极高——这些在游戏/研究 benchmark 中都不存在。
- **缺乏架构选择指南**：开发者面对"用不用 ReAct？单 Agent 还是多 Agent？完整记忆还是摘要记忆？"这些问题时，只能凭直觉。

### AgentArch 的目标

填补上述空白：对 18 种架构配置 × 6 种主流 LLM 在两个真实企业工作流上进行系统评估，目标是提供**有数据支撑的架构决策指南**。

## 评估框架：四维架构空间

AgentArch 构建了一个四维评估空间，每个维度对应一个关键的架构选择：

```
┌─────────────────────────────────────────────────────────────────┐
│                    AgentArch 四维架构空间                         │
├─────────────────────────────────────────────────────────────────┤
│  维度1: Orchestration（编排策略）                                 │
│  ├─ Orchestrator-led, Isolated（集中式，Agent 间不直接通信）      │
│  ├─ Orchestrator-led, Open（集中式，Agent 间直接通信）             │
│  └─ Single Agent（无编排器，单一Agent处理所有请求）                │
├─────────────────────────────────────────────────────────────────┤
│  维度2: Agent Style（Agent 风格）                                 │
│  ├─ Function Calling（直接工具调用，无中间推理）                   │
│  └─ ReAct（显式推理步骤后再选择行动）                             │
├─────────────────────────────────────────────────────────────────┤
│  维度3: Memory（记忆管理）                                        │
│  ├─ Complete Memory（Agent 看到所有历史工具调用和返回）             │
│  └─ Summary Memory（Agent 只收到其他 Agent 的最终摘要）            │
├─────────────────────────────────────────────────────────────────┤
│  维度4: Thinking Tools（思考工具）                                 │
│  ├─ Enabled（提供 math 和 synthesize_collected_info 等推理工具）   │
│  └─ Disabled                                                    │
└─────────────────────────────────────────────────────────────────┘
3 × 2 × 2 × 2 = 18 种配置
```

## 核心发现

### 1. 性能上限：企业任务远比想象中困难

即使是最优配置，模型在企业任务上的表现也相当有限：

| 任务类型 | 最高成功率 | 模型 |
|---------|-----------|------|
| 简单任务（TO） | **70.8%** | Sonnet 4 |
| 复杂任务（CR） | **35.3%** | GPT-4.1 |

**这个数字对工程实践有重要含义**：不要高估当前 Agent 在企业工作流中的可靠性。35% 的复杂任务成功率意味着大约每 3 个请求就有 2 个会失败或需要人工介入——这直接决定了系统的维护成本和用户体验设计方向。

### 2. 最优架构因模型而异——不存在万能设计

这是最反直觉的发现。论文的核心论点是：**没有一种架构在所有模型上都最优**。

具体表现：

- **不同模型偏好的架构不同**：GPT-4.1 在 Orchestrator-led Isolated 配置下表现最好；Sonnet 4 可能偏好不同的组合。
- **大模型的架构鲁棒性更强**：o3-mini（小型推理模型）在不同架构间波动极大（CV 高达 143.7），而 GPT-4.1 的波动相对较小（CV=27.0）。
- **小模型在简单任务上可与大模型匹敌**：在最优架构下，Sonnet 4（中型模型）在 TO 任务上与 GPT-4.1 差距不大——但这完全依赖"选对架构"这个前提。

```
模型一致性分析（CV = 标准差/均值，越低越稳定）：

              TO 任务        CR 任务
GPT-4.1       CV=27.0        CV=34.4     ← 最稳定
Sonnet 4      CV=32.1        CV=79.1
GPT-4.1-mini  CV=57.2        CV=110.7
GPT-4o        CV=59.7        CV=83.9
o3-mini       CV=143.7       CV=104.0    ← 最不稳定
LLaMA 70B     CV=286.8       N/A         ← 几乎完全失效
```

### 3. ReAct 在多 Agent 系统中的严重退化

这是架构层面的关键警示：

> **在多 Agent 系统中使用 ReAct 风格的 Agent 间通信会导致显著的性能下降。**

原因分析：当多个 Agent 都使用 ReAct（先推理再行动）时，推理链在 Agent 间叠加，导致：
- 上下文快速膨胀（每次工具调用都附带推理文本）
- 跨 Agent 的推理链一致性难以保证
- 错误推理的传播被放大

这与单 Agent 场景下 ReAct 通常有效的结论形成鲜明对比——**ReAct 的适用性高度依赖于 Agent 数量和交互复杂度**。

### 4. Memory 策略的权衡

| 策略 | 优势 | 劣势 |
|------|------|------|
| **Complete Memory** | 信息完整，决策质量高 | 上下文快速膨胀，复杂任务下反而降低性能 |
| **Summary Memory** | 控制 token 消耗 | 关键细节可能被丢失，摘要质量依赖模型能力 |

在 TO 简单任务上，Complete Memory 的信息优势能带来收益；但在 CR 复杂任务上，上下文膨胀的代价超过了信息完整的收益。这说明**记忆策略的选择需要基于任务复杂度而非任务类型**。

### 5. Thinking Tools 的条件收益

Thinking Tools（math、synthesize_collected_info）主要在需要**数值计算和多源信息整合**的场景下有价值。对于 TO 任务（涉及日期范围计算和余额比较），启用 Thinking Tools 能显著提升准确性；但对于 CR 任务的分类/路由决策，增加的推理开销可能不值得。

## 对工程师的实用建议

基于以上发现，AgentArch 提供了以下架构决策框架：

### 决策树

```
任务复杂度？
├─ 简单任务（如表单处理、状态查询）
│   └─ 可考虑小模型 + 正确架构，成本大幅降低
│       └─ 优先尝试 Function Calling + Summary Memory
│           └─ 如果涉及计算 → 启用 Thinking Tools
│
└─ 复杂任务（如客户服务、多步协调）
    └─ 建议大模型（GPT-4.1 / Sonnet 4）
        └─ 优先尝试 Orchestrator-led Isolated + Complete Memory
            └─ 避免在多 Agent 场景下使用 ReAct
```

### 不要假设"最新模型 = 最优架构"

o3-mini 的高波动性（CV=143.7）说明：小型推理模型对架构选择极度敏感。用错架构会让 o3-mini 的性能崩溃到接近零。部署这类模型需要更仔细的架构调优和 A/B 测试。

### 企业部署的可靠性预期

如果你的目标是在复杂企业工作流上达到 80% 自动化率，以当前模型能力，这个目标**几乎不可达**（最高 35.3%）。更现实的路线图：

- **简单任务**：目标是 70-80% 自动化率（可行）
- **复杂任务**：目标是 40-50% 自动化率 + 人工兜底机制（更现实）

## 局限性

1. **模型覆盖有限**：只测了 6 个模型（GPT-4.1、GPT-4o、GPT-4.1-mini、o3-mini、LLaMA 70B、Sonnet 4），Claude 系列缺失
2. **任务数量有限**：只有 2 个企业场景（TO 和 CR），虽然评测了 60 个不同请求，但代表性仍然有限
3. **2025年数据**：论文 v1 是 Sep 2025，v2 是 Jan 2026，所测模型能力可能已发生变化

## 结论

AgentArch 的核心贡献不是那个 35.3% 的数字，而是**证明了最优架构因模型而异**这个反直觉的结论。在实际工程中，这意味着：

1. **架构调优是必须的**：不能假设别人有效的架构在你这里也有效
2. **模型选择和架构选择是耦合的**：选模型时就要考虑配套的架构
3. **多 Agent + ReAct 是危险组合**：除非有明确的收益证据，否则默认避免

> 如果你正在为企业场景选型 Agent 架构，建议先用 AgentArch 的四维框架做配置实验，再决定生产部署的组合——而不是直接套用"最佳实践"。
