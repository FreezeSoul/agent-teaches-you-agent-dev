# DAAO：难度感知的多智能体工作流动态编排

> arXiv:2509.11079 | WWW 2026 | AutoAgents-ai | Jinwei Su et al.

---

## 核心问题：多智能体系统的「一刀切」困境

当前多智能体 Agent 框架在编排工作流时面临一个根本性矛盾：

**为所有查询使用统一的工作流 → 要么浪费资源，要么能力不足。**

具体表现两类：

1. **过处理（Over-processing）简单查询**：用一个深度的多阶段 Agent 协作流程处理一个简单的事实性问题，消耗大量 Token 和调用次数，但用户只需要一个答案。

2. **欠处理（Under-processing）复杂查询**：用一个相对简单的串行流程处理多跳推理或代码生成等复杂任务，能力不足以完成任务。

这个问题的根源在于：现有框架（ADAS、AFlow 等）在**任务级别（task-level）** 构建工作流——即为一个完整任务类别（如"所有代码生成任务"）生成一个固定的工作流，而不是为**每个具体查询（query-level）** 动态生成匹配的流程。

> 论文原文给出一个直观例子：用户请求"生成某地旅行指南"，一个只做信息检索和摘要的工作流远远不够；但同样的工作流对"某地天气如何"这个问题就过度了。

DAAO（Difficulty-Aware Agentic Orchestration）提出的核心命题是：**以查询难度为控制信号，动态生成与之匹配的工作流，在精度和效率之间取得最优平衡。**

## 三模块架构：难度估计 → 工作流分配 → 模型路由

DAAO 的设计由三个相互依赖的模块组成，以难度估计值 d ∈ (0,1) 为控制中枢：

```
查询 → [难度估计器 θd] → d
                          ├→ [操作符分配器 θo] → 工作流 G = {V, E}
                          └→ [LLM路由 θm]     → 模型分配 M_i
     ↓
  执行
     ↓
  质量评估 → 成功/失败信号 → 反馈更新 θd, θo, θm
```

### 模块一：难度估计器（Variational Autoencoder）

难度估计器是 DAAO 的核心创新点。它不是使用人工标注的查询难度标签，而是通过 **变分自编码器（VAE）** 从查询的 posterior knowledge 中自动学习难度表示。

**训练方式**：一个 controller 网络以输入查询为条件，采样多智能体架构；然后根据工作流是否成功解决查询，产生反馈信号来更新策略。

**难度即奖励机制（Reward-like Policy）**：
- 工作流成功解决查询 → 降低该查询的难度估计值 → 未来可以使用更简单的工作流
- 工作流失败 → 提高难度估计值 → 触发更复杂、更有能力的工作流

这个机制的本质是：难度不是被"标注"的，而是被"发现"的。如果一个查询用简单工作流就能解决，它就是"简单的"；如果简单工作流总是失败，它就是"困难的"。这是一种**无需人工干预的自动化难度对齐**。

**VAE 的作用**：将查询编码到一个连续潜在空间 z，然后通过一个难度 head 输出标量难度值 d ∈ (0,1)。VAE 的结构保证了相似的查询有相似的潜在表示，从而产生相似的难度估计。

### 模块二：操作符分配器（Operator Allocator）

DAAO 将多智能体工作流建模为 **agentic net**：一个包含所有可能多智能体架构的概率性连续分布。

形式化定义：

```
设 M = 可用LLM集合（如 GPT-4o、Claude、Gemini 等异构模型）
设 S = 协作协议集合（如 Chain-of-Thought、Debate、Ensemble 等）

可行操作符集 O ⊆ M × S

工作流 G = {V, E} 是一个有向无环图（DAG）
V = 节点（操作符）
E = 节点之间的边（依赖关系）
```

**动态工作流生成**：给定查询和难度估计值 d，操作符分配器输出：
- 选择哪些操作符组成工作流节点
- 确定工作流的深度（层数）
- 确定节点之间的连接关系（并行或串行）

难度 d 影响输出的工作流复杂度：d 越低（简单查询）→ 浅层简单工作流；d 越高（困难查询）→ 深层复杂工作流。

### 模块三：成本感知 LLM 路由（Cost-aware LLM Router）

DAAO 解决的另一个关键问题是 **LLM 异构性**：现有大多数多智能体框架使用单一的 backbone 模型（如全部用 GPT-4o-mini），忽略了不同 LLM 在不同任务上的能力-成本差异。

**路由机制**：给定难度 d 和资源约束，LLM 路由为工作流中的不同操作符分配不同的模型：

```
P(M_i | query, d) = softmax(W_r · [h_query; d])
```

直觉上：
- 简单操作符（如事实检索）→ 使用便宜快速的模型
- 复杂操作符（如多跳推理）→ 使用能力强的模型

这使得 DAAO 在整体精度不降的前提下，显著降低推理成本。

## 自适应反馈环：难度估计的持续更新

DAAO 的难度估计不是一次性完成的，而是在**运行时持续更新**：

```
初始化：用 VAE 从查询的语义特征中推断初始难度 d_0
执行：   根据 d_0 生成并执行工作流
评估：   检查输出质量，判断是否成功
更新：   成功 → d ← d - ε（降低难度，简化下次工作流）
        失败 → d ← d + ε（提高难度，下次用更复杂策略）
```

这个机制的关键洞察是：**难度不是查询的固有属性，而是查询与工作流能力之间的相对关系。** 同样的查询，用不同能力的工作流解决，难度感知会不同。这与心理学中的难度概念（任务对特定执行者的相对性）一致。

## 实验结果：精度与效率的双赢

DAAO 在 6 个广泛使用的基准上评估：

| 基准 | 类型 | DAAO 提升（vs 自动编排最优）|
|------|------|--------------------------|
| HumanEval | 代码生成 | +3.5% ∼ +15.2% |
| MBPP | 代码生成 | +3.5% ∼ +15.2% |
| GSM8K | 数学推理 | +3.5% ∼ +15.2% |
| MATH | 数学推理 | +3.5% ∼ +15.2% |
| MMLU | 知识推理 | +3.5% ∼ +15.2% |
| GAIA | 工具使用 | +3.5% ∼ +15.2% |

相比 LLM 路由方法（MasRouter 等），DAAO 提升 3.2% ∼ 10.2%。

**成本效率（MATH 基准）**：
- 训练成本：仅需 SOTA baseline 的 **65%**
- 推理成本：仅需 SOTA baseline 的 **41%**

这说明难度感知的路由在保持精度的同时，显著降低了计算成本。

**泛化性**：DAAO 展现了跨不同 LLM backbone 和跨数据集的强大迁移能力。

## 与编排领域其他研究的互补关系

DAAO 与本知识库中已收录的 VMAO（2603.11445）和 HERA（2604.00901）共同构成自适应多智能体系统的三个互补维度：

| 系统 | 自适应机制 | 核心信号 |
|------|-----------|---------|
| **VMAO** | Plan-Execute-Verify-Replan | 验证结果（Verify）|
| **HERA** | 拓扑演进（增删 Agent）| Profile-Insight-Utility 经验 |
| **DAAO** | 难度路由（工作流 + 模型）| 学习得到的难度估计 |

**VMAO** 解决的是"执行过程中发现计划失败后如何重规划"（反应式适应）；**HERA** 解决的是"长期运行中如何演进系统拓扑"（主动式适应）；**DAAO** 解决的是"如何为每个查询分配合适的初始工作流和计算资源"（前瞻式适应）。三者处于不同的时间粒度和适应维度，组合使用可以构建更完整的自适应多智能体系统。

## 工程可用性评估

### 可用部分

1. **难度估计的 VAE 框架**：核心思路清晰——用成功/失败信号作为 reward 来学习查询难度，无需人工标注。工程实现有 GitHub（AutoAgents-ai/DAAO）可参考。

2. **LLM 异构路由**：成本-性能权衡的模型路由机制有直接工程价值，可以独立于 VAE 难度估计单独使用。

3. **DAG 形式化的工作流表示**：O ⊆ M × S 的形式化定义对工程设计有参考价值。

### 已知局限

1. **VAE 训练成本**：需要额外的训练阶段来学习难度表示，不适合完全没有训练基础设施的团队。

2. **协作协议集合 S 的预定义**：需要预先定义好可用的协作协议（CoT、Debate 等），S 的选择影响最终工作流空间。

3. **仅覆盖协作类协议**：DAAO 主要处理协作类多智能体系统，对竞争性或对抗性多智能体场景未涉及。

4. **难度估计的初始化冷启动**：初始 VAE 需要一个warm-up过程，初期难度估计可能不准确。

### 适用场景

✅ **适合**：对成本敏感的在线推理系统、需要同时优化精度和效率的生产环境、有能力维护模型训练 pipeline 的团队。

❌ **不适合**：对抗性多智能体场景、对响应延迟极度敏感的实时系统、无法进行模型训练的纯推理部署。
