# How Are AI Agents Used? Evidence from 177,000 MCP Tools

> **本质**：首个大规模实证研究，通过追踪 177,436 个 MCP 工具的使用数据，揭示 AI Agent 正在从"感知-推理"转向"行动-修改"，并首次为监管机构提供在模型输出层之外监控 AI 风险的可行方法

**论文**：[arXiv:2603.23802](https://arxiv.org/abs/2603.23802) | **作者**：Merlin Stein | **发表**：2026-03-25 | **评分**：17/20

---

## 一、基本概念

### 研究问题

当前 AI Agent 生态的真实使用状况如何？它们在执行什么类型的任务？监管机构如何在模型层之外监控 AI 风险？

现有监控方法聚焦于 LLM 输出，但 AI Agent 通过**工具层**（Tool Layer）与外部环境交互——这个中间层目前处于监管盲区。

### 研究方法

研究团队通过监测公共 MCP Server 仓库，追踪 **2024 年 11 月至 2026 年 2 月**期间创建的 **177,436 个 Agent 工具**，并结合 NPM 和 PyPI 的下载数据（覆盖 3,854 个 MCP Server，42,498 个工具）来估算实际使用趋势。

**工具分类框架**（三层，按直接影响类型）：

| 类型 | 定义 | 示例 |
|------|------|------|
| **Perception（感知）工具** | 访问和读取外部数据 | 搜索文件、读取数据库、网页爬取 |
| **Reasoning（推理）工具** | 分析数据或概念 | 数据分析、代码审查、内容总结 |
| **Action（行动）工具** | 直接修改外部环境 | 编辑文件、发送邮件、执行金融交易、操控物理无人机 |

**第二维度**（按环境约束程度）：

| 维度 | 定义 | 示例 |
|------|------|------|
| **Constrained（受约束）环境** | 窄范围、有特定 API 的操作 | 执行特定 API 调用、执行固定工作流 |
| **Unconstrained（无约束）环境** | 通用操作，可访问任意环境 | 操作电脑、操控浏览器 |

## 二、核心发现

### 发现一：软件开发生态主导，但并非全部

**67%** 的工具和 **90%** 的下载量来自软件相关领域。这意味着：

- MCP 生态目前主要是一个**开发者工具生态**
- 但金融、行政等领域也在快速增长
- 工具覆盖的职业领域远比主流 AI 评测基准（GAIA/OSWorld）宽广

> 这印证了 AI4Work Benchmark Mismatch 论文的核心结论：当前 Agent 评测严重偏向软件工程，无法反映真实劳动力市场结构。

### 发现二：行动工具从 27% 飙升至 65%——Agent 正在从"观察者"变成"执行者"

这是论文最具趋势性意义的发现。

16 个月内，Action 工具占所有工具使用的比例：

```
2024-11:  27% ████████░░░░░░░░░░░░░░░
2026-02:  65% █████████████████████████
```

**驱动因素**：通用无约束工具的增加——让 Agent 可以直接操控电脑或浏览器，而非通过特定 API 执行受限操作。

这一转变意味着：
- **AI Agent 的能力边界正在快速扩展**：从"给你建议"到"替你执行"
- **风险同步扩大**：能够修改外部环境的 Agent 若出现失误或被滥用，后果远大于纯推理 Agent

### 发现三：行动工具的五重风险属性

论文提出了一个分析框架：行动工具通过以下五个维度扩展 Agent 的行动空间并放大风险：

| # | 维度 | 从 | 到 |
|---|------|-----|-----|
| 1 | 直接影响类型 | 感知工具 | 行动工具 |
| 2 | 环境约束程度 | 受约束环境 | 无约束环境 |
| 3 | 任务 stakes | 低风险任务 | 高风险任务 |
| 4 | 部署规模 | 小范围使用 | 跨领域跨地域广泛使用 |
| 5 | 代码来源 | 人工编写工具 | AI 生成工具 |

### 发现四：金融交易是增长最快的高风险行动领域

虽然大多数行动工具支持中等风险任务（如编辑文件），但**金融交易类工具增长最快**：

> "While most action tools support medium-stakes tasks like editing files, there are action tools for higher-stakes tasks like financial transactions."

这引发了一个关键担忧：

- **Agent 银行挤兑风险**：大量 Agent 同时使用高风险银行业工具进行自动提现，可能引发类似"Agent 银行挤兑"的流动性危机
- **级联失效风险**：如果多个 Agent 基于同一 LLM 运行，一旦该 LLM 出现对齐失败，错误会传播到所有基于它的 Agent

### 发现五：工具层监控——监管机构的新方法论

论文为政府和监管机构提出了一套**在工具层进行 AI 监控**的可行方案：

1. **监测公共 Agent 工具的可用性和使用情况**，预估大规模 Agent 部署的规模
2. **追踪行动工具的比例变化**，识别 Agent 能力边界的扩张速度
3. **识别高风险工具领域**（如金融交易工具），提前准备监管框架

> 这与英国政府金融监管机构合作开展了试点监测项目。

## 三、工具生态的量化和趋势

### 按下载量的使用分布

| 类别 | 占总下载量比例 |
|------|--------------|
| 软件开发工具 | ~90% |
| 其他领域 | ~10% |

软件开发工具的绝对主导地位反映了当前 MCP 生态的**开发者驱动**本质。

### 感知/推理/行动工具的使用趋势

| 时间点 | 感知 | 推理 | 行动 |
|--------|------|------|------|
| 2024-11 | 约 45% | 约 28% | **27%** |
| 2026-02 | 约 15% | 约 20% | **65%** |

行动工具在 16 个月内占比翻倍以上增长，是 MCP 生态最显著的结构性变化。

## 五、局限性

1. **数据来源局限**：仅监测公共 MCP Server 仓库，私有/企业 MCP 部署不在统计范围内
2. **下载量≠实际使用**：NPM/PyPI 下载量可能包含开发测试而非生产使用
3. **时间窗口有限**：16 个月的数据可能无法完全反映长期趋势
4. **工具分类依赖 LLM 辅助**：感知/推理/行动的三分类由 LLM 辅助完成，存在标注噪声
5. **O*NET 映射的局限**：O*NET 是美国职业分类体系，对其他国家的适用性有限

## 六、实践启示

### 对开发者
- **工具设计决策**：如果你的工具是 Action 类型的，需要考虑额外的安全措施（沙箱、确认流程）
- **无约束 vs 受约束**：将工具设计为受约束 API 比通用浏览器操控更安全
- **监控工具选择**：参考本文的分类框架，评估你的工具对用户造成的直接风险程度

### 对企业/安全团队
- **工具层监控**：除了监控 LLM 输出，还应该监控 Agent 调用了哪些工具、执行了什么操作
- **高风险工具识别**：金融交易、数据删除、系统配置类工具应视为高风险，建立审批流程
- **供应链风险**：如果你的 Agent 依赖第三方 MCP Server，需要评估该 Server 的工具层安全状况

### 对监管机构
- **工具层监管框架**：本研究提供了在模型层之外监控 AI 风险的可行方法论
- **高风险工具登记**：建立 MCP 行动工具的分类登记制度，特别是金融、医疗、基础设施相关工具
- **跨境协作**：由于 MCP 工具的跨境使用特性，需要国际协调的监管框架
